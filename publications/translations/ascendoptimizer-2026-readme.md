# Интересные особенности AscendOptimizer

Заметки по статье: [AscendOptimizer: агент с эпизодическим опытом для оптимизации операторов Ascend NPU](ascendoptimizer-2026.md).

## Двухконтурная оптимизация

AscendOptimizer разделяет работу на две разные задачи: сначала улучшает структуру device-side ядра, затем подбирает host-side тайлинг для уже улучшенного ядра. Это важно для AscendC, потому что производительность оператора зависит не только от кода ядра, но и от того, как хостовая часть разбивает данные и организует их подачу.

> Конкретно, у AscendOptimizer есть офлайн-фаза bootstrap-построения опыта по всему бенчмарку и онлайн-фаза оптимизации для каждого оператора. Офлайн каждый оператор исследуется с помощью Optimization Rewind при фиксированном бюджете rewind. После завершения rewind по всему бенчмарку аппаратно подтвержденные пары деоптимизации сводятся в общий банк опыта. Онлайн стадия I использует этот банк для улучшения целевого ядра, а стадия II затем ищет конфигурации хостового тайлинга для уже улучшенного ядра.

Комментарий: **конфигурации тайлинга подбираются по обратной связи от реального оборудования**. То есть система не пытается заранее вывести все правила тайлинга из документации или статического анализа, а компилирует, запускает и профилирует кандидатов на Ascend NPU.

## Банк опыта вместо дообучения

Система не обновляет веса LLM через SFT или RL. Вместо этого она строит внешний банк опыта: записи о том, какие оптимизационные мотивы были удалены, как это повлияло на производительность и какое обратное переписывание можно использовать позже.

## Optimization Rewind

Идея Rewind необычна: система не только пытается улучшать код, но и контролируемо ухудшает сильные реализации. Если удаление мотива замедляет оператор и при этом сохраняет корректность, этот мотив считается полезным и превращается в переиспользуемую оптимизационную подсказку.

## Разные механизмы для ядра и тайлинга

Для ядра работает перенос структурных паттернов через retrieval из банка опыта. Для тайлинга перенос слабее, потому что допустимость зависит от формы входов, выравнивания, емкости буферов и ограничений железа. Поэтому тайлинг ищется эволюционно, с проверкой кандидатов на устройстве.

## Практическая метрика бюджета

Каждая оценка кандидата включает компиляцию, проверку корректности и профилирование. Авторы явно считают бюджет таких аппаратных оценок, потому что каждая попытка стоит реального wall-clock времени.
